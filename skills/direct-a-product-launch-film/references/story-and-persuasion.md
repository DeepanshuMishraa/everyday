## Product truth

Start with evidence, not adjectives. Gather screenshots, working interactions, source copy, installation commands, real outputs, metrics with provenance, and constraints. Sort each proposed claim into:

- **Demonstrable** — visible in the product or supported by supplied evidence.
- **Plausible but unproven** — useful as internal direction, unsafe as launch copy.
- **Aspirational** — exclude unless the user explicitly labels it as vision.

Use demonstrable claims in the film. If the product is early, make the value specific rather than inflated.

## Launch-film argument

Write the film as one sentence:

> For **audience**, who struggles with **problem**, **product** provides **promise**, proven by **visible loop**, so take **action**.

Every beat must serve one clause. Remove any scene that is merely attractive.

## Hook patterns

Choose one:

- **Audience language** — a question or frustration the viewer already says.
- **Broken default** — expose an accepted interaction that should be better.
- **Immediate outcome** — show the finished result, then reveal how.
- **Transformation** — old state physically becomes the new state.
- **Command** — a short kinetic invitation suited to developer tools.

Avoid “Introducing…”, generic superlatives, and abstract mission statements unless brand recognition already carries the hook.

The hook should be understandable without knowing the product name.

## Proof design

Use a complete micro-loop:

1. user intent;
2. product response;
3. user action;
4. resolved state.

A search field without a result is not proof. A terminal command without a receipt is not proof. A question UI without selection and progress is not proof.

Prefer one loop shown clearly over three features shown partially.

## Beat responsibilities

| Beat | Must answer | Common failure |
|---|---|---|
| Hook | Why should I care? | Starts with company language |
| Reveal | What is this? | Logo appears without a promise |
| Proof | Does it really work? | Decorative UI montage |
| Payoff | Why is it better? | Feature named without consequence |
| Action | What do I do now? | Tiny URL or competing CTAs |

## Storyboard fields

For every scene record:

- global start and end;
- local duration;
- narrative role;
- exact on-screen copy;
- product evidence used;
- focal element and secondary focal point;
- build, breathe, resolve timings;
- music cue;
- transition in;
- poster timestamp;
- acceptance test.

Acceptance tests should be observable: “selected answer changes color and progress advances,” not “feels intuitive.”

## Copy rules

- Use one main sentence per frame.
- Prefer verbs and outcomes to category labels.
- Set product names, URLs, and commands exactly.
- Keep feed-size readability in mind; viewers often see the film below full screen.
- Do not make the viewer read body copy while fast motion is occurring.
- End with one primary CTA. Secondary metadata may support it but must not compete.

## Duration pressure

Allocate proportions before seconds:

- hook: 10–18%;
- reveal: 15–22%;
- proof: 25–40%;
- payoff: 15–25%;
- action: 10–20%.

Compress or combine beats for bumpers. Expand proof and comprehension holds for explainers. Move boundaries to musical phrases, narration clauses, and product complexity. Preserve enough time to read the final action at the target viewing size.

Do not stretch a weak idea to fill a long runtime. Do not cram a multi-step product into a short runtime merely to imitate a fashionable launch-film length.
