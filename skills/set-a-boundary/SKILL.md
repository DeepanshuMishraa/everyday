# Set a Boundary

## Interaction

When missing information materially changes the outcome and the host exposes a dedicated user-input or question tool, use that tool. Ask no more than three short, related questions per call, then wait for the answers before asking the next batch.

Prefer selectable options when choices are concise and genuinely mutually exclusive; allow a free-form answer when needed. Do not present a long questionnaire in normal chat. If no native input tool is available, ask one concise blocking question at a time. For non-blocking gaps, state the assumption and continue.

A boundary names what the user will do in a specific situation. It is not a rule that forces another person to change.

## Define it

Ask about the relationship, recurring behaviour, what the user wants to protect, what remains acceptable, practical dependencies, likely pushback, and any safety or power imbalance.

Write the boundary as:

> When <observable situation>, I will <action within my control>.

Make the follow-through safe, proportionate, and realistic. A consequence the user cannot or will not carry out weakens the boundary.

## Say it once

Build a short script with:

1. optional context
2. the limit
3. what the user will do
4. any next practical step

Use [BOUNDARY-SCRIPTS.md](BOUNDARY-SCRIPTS.md) for patterns. Remove accusations, diagnoses, courtroom evidence, and requests for permission.

## Rehearse pressure

Practice the most likely response: bargaining, guilt, anger, minimising, repeated questions, or apparent agreement without changed behaviour. Use a brief broken-record reply when the limit has already been understood.

## Plan delivery and review

Choose a safe time, channel, privacy level, and support person. Review whether the user communicated and followed the boundary—not whether the other person approved it.

For coercion, abuse, stalking, retaliation, or immediate danger, leave ordinary rehearsal and prioritise specialist safety support. Do not pressure confrontation or continued contact.
